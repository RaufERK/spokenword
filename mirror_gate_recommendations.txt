MIRROR GATE — РЕКОМЕНДАЦИИ ПО УЛУЧШЕНИЮ ПЛАНА И ТОНКИЕ МЕСТА

Дата: 2026-04-09

==================================================
1. ОБЩАЯ ОЦЕНКА
==================================================
План сильный и реалистичный.
Базовая идея правильная:
- PRIMARY в РФ
- EU как зеркало и proxy
- единый источник истины по данным
- видео как файлы, а не в БД

Это хорошая production-схема для текущего масштаба проекта.

==================================================
2. ЧТО СТОИТ УЛУЧШИТЬ В ПЛАНЕ
==================================================

2.1. ЯВНО ЗАФИКСИРОВАТЬ ОДИН WRITE-MASTER
Нужно прямо прописать:
- запись в БД только в РФ
- создание новых пользователей только в РФ
- чат пишет только в РФ
- обновление оплат / доступов только в РФ

Это важно, чтобы потом случайно не начать писать данные и на EU тоже.

--------------------------------------------------

2.2. ДОБАВИТЬ РАЗДЕЛ ПРО COOKIE / SESSION / AUTH
Это одно из самых опасных мест.

Нужно проверить:
- какой domain у cookie
- поддерживает ли он и www.spoken-word.ru, и eu.spoken-word.ru
- выставлен ли Secure
- какой SameSite
- не зашит ли абсолютный callback URL на один домен

Практический совет:
- использовать общий cookie domain: .spoken-word.ru
- всё держать только по HTTPS
- заранее протестировать логин на обоих доменах

--------------------------------------------------

2.3. ПРОВЕРИТЬ, НЕТ ЛИ В КОДЕ АБСОЛЮТНЫХ URL
Опасные места:
- fetch('https://www.spoken-word.ru/api/...')
- ссылки в auth
- ссылки в reset password
- ссылки в email-уведомлениях
- пути к видео и превью
- websocket URL
- upload-service URL

Лучше везде:
- относительные /api/...
- или env-переменные по домену

--------------------------------------------------

2.4. ДОБАВИТЬ ЯВНУЮ ПРОВЕРКУ WEBSOCKET / SSE / LONG POLLING
Если чат realtime, надо заранее понять:
- это WebSocket?
- SSE?
- polling?

Если WebSocket:
- EU nginx должен корректно проксировать Upgrade/Connection
- таймауты должны быть адекватные
- иначе чат будет "иногда работать, иногда нет"

--------------------------------------------------

2.5. ДОБАВИТЬ ПРОВЕРКУ ЗАГРУЗКИ ФАЙЛОВ
Если в проекте есть загрузки:
- аватары
- файлы
- видео
- платный контент

Нужно заранее понять:
- они принимаются только в РФ?
- не остаются ли временные файлы на EU?
- не зашиты ли пути на старый сервер?

--------------------------------------------------

2.6. SQLITE ОСТАВИТЬ ВРЕМЕННО, НО ПОМЕТИТЬ КАК РИСК
SQLite на текущем этапе может жить.
Но нужно честно пометить в плане:
- это временный компромисс
- при росте активности лучше перейти на PostgreSQL

Особенно если одновременно будут:
- чат
- регистрации
- оплаты
- админские изменения

--------------------------------------------------

2.7. ДОБАВИТЬ ЧЕК-ЛИСТ БЭКАПОВ
Нужно явно указать:
- где лежит основной backup
- как часто делается
- как проверить восстановление
- кто отвечает за проверку backup
- сколько дней хранить копии

Принцип:
backup, который ни разу не восстанавливали, не считается надёжным.

--------------------------------------------------

2.8. ДОБАВИТЬ ROLLBACK-ПЛАН
Очень желательно прописать:
- как откатить DNS обратно на EU
- как вернуть EU в роль PRIMARY при неудаче
- как быстро отключить зеркало
- какие данные считаются потерянными при откате

Без rollback-плана миграция нервирует сильнее, чем должна.

==================================================
3. ТОНКИЕ МЕСТА: ГДЕ МОЖЕТ СЛОМАТЬСЯ
==================================================

3.1. LOG IN / LOG OUT / SESSION
Самый частый сбой:
- пользователь логинится на одном домене
- на другом домене сессия не видна
- logout очищает cookie не там
- редирект после логина уводит на другой хост

Что проверить:
- вход
- выход
- refresh страницы
- вход с обычного окна и инкогнито
- работа на обоих доменах

--------------------------------------------------

3.2. ЧАТ REALTIME
Может сломаться так:
- сообщения отправляются
- но новые сообщения не приходят без refresh
- websocket рвётся через proxy
- polling кэшируется или таймаутится

Что проверить:
- 2 клиента на РФ домене
- 2 клиента на EU домене
- 1 клиент в РФ + 1 клиент в EU
- сообщение должно появляться у всех

--------------------------------------------------

3.3. ABSOLUTE URL В НЕОЖИДАННЫХ МЕСТАХ
Часто ломается не в основном коде, а в:
- middleware
- auth callbacks
- SEO canonical
- Open Graph image URL
- sitemap
- robots.txt
- email templates
- upload-service
- ffmpeg/output paths
- admin links

--------------------------------------------------

3.4. COOKIE DOMAIN И SAME_SITE
Если будет неверно:
- логин на EU не сохранится
- формы начнут "выбрасывать"
- csrf может ломаться
- запросы к API будут идти без авторизации

--------------------------------------------------

3.5. ПРОКСИРОВАНИЕ /API НА EU
Может сломаться:
- заголовки Host / X-Forwarded-For
- большие request body
- upload timeout
- длинные запросы
- websocket upgrade

Что проверить:
- login
- chat send
- payment update
- profile edit
- admin actions

--------------------------------------------------

3.6. SSL / CERTIFICATES
Отдельно проверить:
- сертификат на www.spoken-word.ru после перевода в РФ
- сертификат на eu.spoken-word.ru
- auto-renew
- не завязана ли выдача сертификата на старую инфраструктуру

--------------------------------------------------

3.7. ВИДЕО И ПУТИ К НИМ
Может сломаться:
- пути зашиты на старый absolute URL
- в EU нет файла
- nginx ищет не в той папке
- range requests работают криво
- mp4 не мотается по таймлайну

Что проверить:
- воспроизведение
- перемотка
- разные браузеры
- fallback на РФ, если файла нет локально

--------------------------------------------------

3.8. DNS И КЭШИ
Даже после смены DNS:
- у части пользователей ещё старый IP
- у провайдера кэш
- локальный DNS cache
- браузерный cache

Поэтому в день переключения нужна спокойная коммуникация:
- у части людей изменения приходят не мгновенно

--------------------------------------------------

3.9. FIREWALL / ACCESS RULES
Можно случайно:
- закрыть нужный порт
- не открыть доступ между EU и РФ
- заблокировать Redis / app / ssh
- ограничить себя самого

Нужно до миграции проверить:
- ssh доступ
- app порт локально
- nginx наружу
- EU → РФ API доступ

--------------------------------------------------

3.10. ДИСК И МЕСТО ПОД ВИДЕО
Очень практичный риск:
- видеоархив влезает сейчас
- но после копий и логов место заканчивается быстрее, чем кажется

Нужно держать контроль:
- свободное место
- рост архива
- логи nginx / pm2
- временные файлы

==================================================
4. ЧТО ДОБАВИТЬ В ДЕНЬ ПЕРЕКЛЮЧЕНИЯ
==================================================

Перед переключением:
- понизить TTL
- проверить сайт по IP РФ
- проверить SSL
- проверить логин
- проверить видео
- проверить чат
- проверить оплату / доступы
- проверить резервную копию БД
- убедиться, что есть rollback-план

Во время переключения:
- заморозить записи на EU
- сделать финальную копию БД
- перезапустить РФ
- переключить DNS
- пройти smoke test

После переключения:
- проверить сайт из РФ
- проверить сайт с EU домена
- проверить чат с двух доменов
- проверить логин
- проверить логи nginx/pm2
- проверить диск и нагрузку

==================================================
5. МИНИМАЛЬНЫЙ SMOKE TEST
==================================================
1. Открывается главная
2. Работает логин
3. Работает logout
4. Работает регистрация
5. Отправляется сообщение в чат
6. Сообщение видно на другом клиенте
7. Видео открывается
8. Видео мотается
9. Открывается платный контент
10. Админка выполняет критичное действие

==================================================
6. МОЙ ИТОГОВЫЙ СОВЕТ
==================================================
План хороший и уже пригоден к реализации.

Главное:
- не усложнять
- держать один write-master
- отдельно проверить auth/cookies/chat/proxy
- обязательно иметь rollback
- не забыть о тесте после реального переключения

==================================================
7. ПРО НАЗВАНИЕ MIRROR GATE
==================================================
MIRROR GATE звучит неплохо.

Как воспринимается:
- Mirror = зеркало, отражение, дублирующий контур
- Gate = ворота, точка входа, шлюз

Общий смысл:
- "ворота к зеркалу"
- "зеркальный шлюз"
- "вход в резервный контур"

По звучанию:
- технично
- немного поэтично
- запоминается
- не выглядит смешно

Но есть нюанс:
название больше похоже на внутреннее кодовое имя проекта, чем на официальное продуктовое название.
Для внутреннего проекта — хорошо.
Для публичного бренда — слишком инженерно.

Если нужен именно внутренний codename:
MIRROR GATE — норм.

Если хочется чуть более строго и инженерно, можно подумать о вариантах:
- RF PRIMARY
- RU PRIMARY / EU MIRROR
- SPOKEN-WORD MIRROR
- DUAL EDGE
- NORTH BRIDGE
- EAST GATE

Если хочется оставить нынешнее имя:
оставляй смело.
Оно не лажа.
