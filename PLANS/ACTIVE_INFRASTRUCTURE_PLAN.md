# Active Infrastructure Plan

Дата: 2026-04-10

Этот файл собирает только те задачи, которые еще реально актуальны после завершения переезда основного `www/spoken-word.ru` на РФ.

## 1. Текущее состояние

### 1.1. Основной прод

- `www.spoken-word.ru` -> РФ
- `spoken-word.ru` -> РФ
- write-master один: РФ

### 1.2. Аудио

- `audio.spoken-word.ru` сейчас снова смотрит на EU `185.200.178.73`
- публичные резолверы `1.1.1.1`, `8.8.8.8`, `9.9.9.9` уже отдают EU IP
- сам сервис `audio-word` на EU отвечает `200`
- `/health` на EU отвечает `ok`
- сертификат на EU для `audio.spoken-word.ru` валиден

Вывод:

- `audio.spoken-word.ru` сейчас считаем рабочим
- переносить его на РФ прямо сейчас не нужно
- `Streamster` пока не трогаем
- RTMP ingest пока остается на EU:
  - `rtmp://185.200.178.73/live`
  - key: `main`

## 2. Срочно

### 2.1. Контрольный backup текущего РФ master

Что это означает:

- снять чистую контрольную копию текущего `prisma/data.db`
- сохранить counts по основным таблицам
- отдельно зафиксировать, что медиа на РФ на месте
- использовать это как точку возврата после успешного cutover и серии правок

Практически это:

1. Скопировать `prisma/data.db` в именованный backup-файл
2. Снять counts по ключевым таблицам
3. Сверить количество архивных видео и paid-content
4. Записать дату и состояние в короткий `.md`

Оценка: 30-60 минут

### 2.2. Дать продакшену стабилизироваться

Смысл:

- не трогать основной прод новыми рискованными переносами подряд
- посмотреть несколько дней на реальную нагрузку, логи, оплаты, чат, архив и загрузки
- не смешивать это окно со сложной инфраструктурной миграцией

Рекомендуемое окно: 2-5 дней спокойной эксплуатации

## 3. Короткий горизонт

### 3.1. International mirror на `spoken-word.info`

Статус:

- основной прод уже в РФ
- для иностранных пользователей домен без `.ru` выглядит лучше, чем `eu.spoken-word.ru`
- как fallback-хост его еще полезно держать 1-2 недели

Рекомендуемая схема:

1. Использовать `spoken-word.info` как публичный international entrypoint
2. На EU оставить локальные видео
3. `/api/*` проксировать в РФ
4. Не допускать самостоятельной записи в отдельную EU-базу
5. Проверить cookie/auth и относительные URL для второго публичного домена

Что проверить отдельно:

- DNS `spoken-word.info`
- SSL для `spoken-word.info`
- `NEXTAUTH`/cookie поведение на `.info`
- доступность сайта для иностранных пользователей

Оценка: 0.5-1 день

### 3.2. Небольшие инфраструктурные хвосты

- health-check/alerting по `nginx`, `pm2`, диску и SSL
- cleanup временных файлов загрузки
- retry для failed compression jobs
- thumbnails/preview generation для видео

Оценка: 0.5-1.5 дня суммарно, можно делать по частям

## 4. Главный следующий технический этап

### 4.1. Миграция SQLite -> PostgreSQL

Статус:

- это не аварийная задача на сегодня
- это самый важный следующий технический проект после стабилизации РФ-прода
- подробный пошаговый план вынесен в `PLANS/POSTGRESQL_MIGRATION_PLAN.md`

Почему это действительно следующий большой шаг:

- сейчас один write-master в РФ, и это уже сильно лучше прежней схемы
- но SQLite остается файлом с ограничениями по конкурентной записи
- чат, платежи, регистрации, доступы и админские операции со временем будут упираться в эти ограничения
- с PostgreSQL будет проще и безопаснее делать дальнейшую схему `RF primary + EU mirror`

Что входит в работу:

1. Перевести Prisma datasource на PostgreSQL
2. Поднять PostgreSQL на РФ
3. Написать и прогнать миграцию данных `SQLite -> PostgreSQL`
4. Проверить ключевые таблицы:
   - `User`
   - `Event`
   - `UserEventAccess`
   - `ChatMessage`
   - `ConferenceFile`
   - `ClassFile`
   - `ContentPackage`
   - `PackageItem`
   - `UserPackageAccess`
   - `StreamLink`
   - `ClassStreamLink`
5. Проверить приложение локально/на тестовом контуре
6. Сделать финальный cutover с rollback-планом

Реалистичная оценка:

- 2 полных сфокусированных дня — да, это реальная оценка

Разбивка:

#### День 1

- подготовка схемы и `.env`
- локальный PostgreSQL
- миграционный скрипт
- сверка counts по таблицам
- локальная проверка логина, чата, оплат, архива, классов, пакетов

#### День 2

- установка PostgreSQL на РФ
- dry-run миграции на сервере
- smoke test
- финальный перенос данных
- переключение приложения на PostgreSQL
- post-cutover проверка

Запас по риску:

- если хотим сделать совсем спокойно, с буфером и отдельной репетицией rollback, лучше считать 2-3 дня

Важно:

- не совмещать миграцию на PostgreSQL с переносом `audio-word`
- не смешивать это в один день с другими инфраструктурными изменениями

## 5. Приоритеты

Приоритет на сейчас:

1. снять контрольный backup РФ
2. дать прод-системе немного стабилизироваться
3. подготовить `spoken-word.info` как international mirror на EU
4. после этого брать отдельным блоком PostgreSQL

## 6. Что уже не является активным планом

Закрыто и больше не требует отдельных планов:

- RF cutover для `www/spoken-word.ru`
- выпуск SSL для `www` и голого домена на РФ
- разворачивание основного прод-контура на РФ
- базовая массовая выдача доступа
- исправление открытия архивных видео на РФ
- срочный фикс `audio.spoken-word.ru` после возврата DNS на EU
