# PostgreSQL Migration Plan

Дата: 2026-04-10

## Цель

Перевести `spokenword` с SQLite на PostgreSQL без спешки, после стабилизации РФ-прода, сохранив понятный rollback.

## Почему это следующий большой шаг

- сейчас `www/spoken-word.ru` уже работает из РФ
- write-master один, это хорошо
- но SQLite остается файлом с ограничениями по конкурентной записи
- чат, оплаты, регистрации, доступы и админские действия будут со временем упираться в эти ограничения
- PostgreSQL нужен не потому что "модно", а потому что он даст надежную базу под дальнейшую схему `RF primary + EU/info mirror`

## Что не делаем в тот же день

- не переносим `audio-word`
- не делаем новые доменные переезды
- не смешиваем с большим UI-рефакторингом

## Реалистичная оценка

- 2 полных сфокусированных дня
- если хотим совсем спокойно, с дополнительной репетицией rollback: 2-3 дня

## Подготовка до старта

1. Зафиксировать quiet period без лишних деплоев
2. Снять контрольный backup текущего РФ SQLite master
3. Подтвердить список таблиц для миграции
4. Подтвердить, что на момент cutover писать будет только РФ

## Таблицы, которые нужно перенести

- `User`
- `Event`
- `UserEventAccess`
- `ChatMessage`
- `ConferenceFile`
- `ClassFile`
- `StreamLink`
- `ClassStreamLink`
- `ContentPackage`
- `PackageItem`
- `UserPackageAccess`

## День 1

### 1. Перевести кодовую базу на поддержку PostgreSQL

1. Обновить datasource в `prisma/schema.prisma`
2. Подготовить `DATABASE_URL` под PostgreSQL
3. Локально поднять PostgreSQL
4. Создать пустую схему Prisma в PostgreSQL

### 2. Написать миграционный скрипт SQLite -> PostgreSQL

Требование к скрипту:

- читает из SQLite
- пишет в PostgreSQL
- переносит данные в корректном порядке по зависимостям
- не теряет `id`
- не искажает даты
- не пересчитывает бизнес-логику на лету

Порядок переноса:

1. `User`
2. `Event`
3. `UserEventAccess`
4. `ContentPackage`
5. `PackageItem`
6. `UserPackageAccess`
7. `ConferenceFile`
8. `ClassFile`
9. `StreamLink`
10. `ClassStreamLink`
11. `ChatMessage`

### 3. Локальная сверка

После миграции проверить:

- count по каждой таблице
- несколько выборочных записей вручную
- оплату и `accessUntil`
- доступ к пакетам
- сообщения чата
- архивные записи и метаданные

### 4. Локальный smoke test приложения

Проверить:

- логин
- логаут
- регистрация
- чат
- архив
- класс
- платные пакеты
- админку

Результат дня 1:

- код работает с PostgreSQL локально
- миграционный скрипт готов
- есть сверка данных

## День 2

### 1. Поднять PostgreSQL на РФ

1. Установить PostgreSQL на РФ
2. Создать пользователя и БД
3. Настроить `DATABASE_URL`
4. Создать пустую схему

### 2. Dry-run на сервере

1. Взять свежую копию `data.db`
2. Прогнать миграционный скрипт в PostgreSQL на РФ
3. Сверить count по таблицам
4. Запустить приложение на PostgreSQL
5. Пройти smoke test на сервере

### 3. Финальный cutover

1. Согласовать короткое окно работ
2. Снять финальную SQLite копию
3. Повторно прогнать миграционный скрипт
4. Переключить приложение на PostgreSQL
5. Перезапустить процессы
6. Пройти финальный smoke test

## Smoke test после cutover

Проверить:

- главная страница
- логин/логаут
- регистрация
- отправка сообщения в чат
- отображение сообщений
- доступы пользователей
- архив и просмотр видео
- класс и стрим-ссылки
- пакеты и доступ к ним
- админские действия по пользователям
- загрузка файла

## Rollback

Если что-то идет не так:

1. Остановить текущий релиз
2. Вернуть приложение на SQLite `DATABASE_URL`
3. Поднять сервисы обратно на SQLite
4. Использовать контрольный backup `data.db`
5. Разобрать проблему отдельно, без давления продакшена

## Definition of Done

- приложение работает на PostgreSQL в РФ
- count по ключевым таблицам совпадает
- логин, чат, доступы, архив, класс, пакеты и загрузка работают
- rollback-путь понятен и проверен
- после перехода не осталось скрытой зависимости от SQLite

## Рекомендация по запуску

Лучший сценарий:

1. Сейчас не трогаем БД
2. Даем продакшену немного спокойно пожить
3. Выделяем отдельные 2 дня только под PostgreSQL
4. После этого уже идем к схеме `spoken-word.info` / EU mirror поверх стабильного PostgreSQL master
